#### 1 在“AI Resume Analyzer”项目中，你具体用Zustand管理了哪些全局状态？并解释为什么选择Zustand而不是React自带的Context API或useReducer。

>在 AI Resume Analyzer 里，我用 Zustand 管的全局状态主要是三类：
>
>系统状态
>
>isLoading、error：全局加载与错误态
认证状态
auth.user、auth.isAuthenticated
signIn / signOut / refreshUser / checkAuthStatus 这些认证动作
跨页面共享的能力对象
fs：文件上传/读取/删除
ai：聊天与简历反馈分析
kv：简历结果的 KV 存取
我选 Zustand 而不是 Context API 或 useReducer，核心原因是：
>
>更少样板代码：不需要写很多 Provider、action type、switch，开发效率更高。
订阅粒度更细：组件可以只订阅自己关心的 slice，减少不必要重渲染。
业务更贴合：我们这个项目有大量“异步副作用 + 跨页面共享能力”（认证、文件、AI、KV），Zustand 把“状态 + 动作”放在一个轻量 store 里更清晰。
可维护性更好：相比 Context 在状态变大时容易臃肿，Zustand 更容易按模块拆分和扩展。


#### 2.你的简历里提到有Vue和React的双框架经验，能简单说说你在Vue和React之间切换时，感受最大的差异是什么吗？这算是我额外好奇的一个问题，你可以简单聊聊

>最大的感受其实集中在数据响应式思维和写法心智模型这两块，切换时需要快速调整思路：
第一，数据驱动的底层逻辑完全不同。Vue 是侵入式的自动响应式，不管是对象还是数组，直接修改数据视图就会自动更新，依赖追踪是框架帮我做的，写起来很省心；但 React 是不可变数据 + 手动触发更新，必须用setState/useState，不能直接改原数据，还要注意引用不变导致不渲染的问题。刚从 Vue 切 React 时，总忍不住直接改对象，后来才慢慢养成不可变更新的习惯。
第二，视图写法的心智模式不一样。Vue 用的是模板语法，template和逻辑分离，贴近原生 HTML，写条件、循环用v-if/v-for，很直观；React 是 JSX 一体化，把结构写在 JS 里，循环用map、条件用三元 / 逻辑运算符，逻辑和视图强绑定。一开始会觉得模板更清晰，后来用惯 JSX，又觉得它处理复杂逻辑、动态渲染更灵活。

#### 3. scss
SCSS 本质上是 CSS 的预处理器，主要解决原生 CSS 在大型项目里可维护性差的问题。它支持嵌套、变量、复用和模块化，让样式从“堆规则”变成“可组织的代码”

>我把 SCSS 当成工程化 CSS 来用，核心价值是统一和可维护。
第一点是全局变量统一主题和组件库风格。我们项目会先在全局层定义主题色、文字色、状态色这些设计令牌，然后页面和组件都复用这一套，不会每个人自己写一套颜色。这样做的好处是，风格一致、改版成本低，比如后面要从一种主题换到另一种主题，改变量就能全局生效，不需要逐页找样式。对组件库也一样，可以基于统一变量做二次定制，保证业务页面和组件库视觉是同一套语言。
>
>第二点是用嵌套加 scoped 提升可读性并隔离样式。SCSS 的嵌套可以把父子结构、状态样式、伪类写在一起，读起来就是组件结构本身，维护时定位很快。再配合 scoped，样式作用域默认限制在当前组件，基本不会出现一个页面改样式把另一个页面误伤的问题。对于多人协作，这点很重要，能明显减少样式冲突和回归问题。

#### 5. 你是如何封装这个useFetch函数的？有什么好处？
>我在项目里封装 useFetch 的思路是，统一所有接口请求的共性流程。
它接收一个 fetchFunction，然后内部维护 data、loading、error 三个状态。执行请求时会先置 loading、清空 error；成功就写入 data；失败就捕获错误；最后在 finally 里恢复 loading，保证状态收口一致。
>
>
>这样做的好处是三点：
第一，减少重复代码，每个页面不用反复写 try/catch/finally；
第二，状态和错误处理风格统一，体验更稳定；
第三，后续要加重试、超时、取消请求时，只需要改 Hook 一处，维护成本很低。
