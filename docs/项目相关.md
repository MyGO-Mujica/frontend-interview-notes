##### 1. 封装 Axios 请求层，统⼀ Token 注⼊、错误处理与重试机制，构建标准化接⼝调⽤模型，降低重复代码与异常分散问题

第一层：先问“为什么这么做”

- 问法：你为什么要封装 Axios 请求层？
> 因为业务接口调用非常分散，如果每个页面都自己写 token、错误提示、401 处理，就会重复、易漏、逻辑不一致。我把它集中到一个请求层里，统一注入 token、统一错误处理和跳转逻辑，让页面只关心业务数据，减少重复和异常分散。

- 你说的“好处”具体体现在哪？
> 主要是三点：
>
>1. 减少重复代码，接口调用更简洁；
>2. 统一异常处理，避免不同页面提示不一致；
>3. 便于项目推进，比如后续加重试、日志、埋点，都在一处改动。

第二层：再问“做了哪些**细节**”

- 问法：Token 是怎么统一注入的？
> 我在请求拦截器里读取 Pinia 的用户仓库 token，写入 Authorization 头，确保所有请求一致携带。

- 问法：401 怎么处理？为什么要做防重复提示？
> 响应拦截器里如果是 401，就提示用户、清空 token、跳转登录。我加了 isRedirecting 标志，避免并发请求导致多次弹窗/多次跳转。

- 重试怎么做的？为什么只对某些请求重试？
> 我只对幂等请求（GET/HEAD/OPTIONS）且网络错误或 5xx 才重试，默认 2 次，指数退避（500ms、1000ms…），避免对 POST 这类非幂等请求造成重复提交。

- 问法：为什么不用对业务失败重试？
> 业务失败通常是参数、权限或业务规则问题，重试没有意义，反而增加压力，所以只对网络或服务端不稳定才重试。


总结：
> 我在项目里把 Axios 封装成统一请求层，核心是把“横切逻辑”集中处理。原因是页面接口太多，如果每个页面都写 token 注入、错误提示、401 跳转，代码重复又容易漏。我在请求拦截器里从 Pinia 取 token 统一写到 Authorization，响应拦截器里统一处理业务失败提示、401 清 token 并跳转登录，还用标志避免重复提示。为了提升稳定性，我补了重试机制：只对幂等请求、网络错误或 5xx 做最多 2 次重试。这样页面层只关心业务数据，异常处理和稳定性策略都在一个地方，后续要扩展日志、埋点也很方便。

##### 2. 梳理⾸屏加载链路，通过路由级懒加载、Vite ⾃动分包与静态资源按需加载进⾏性能优化，核⼼⻚⾯ LCP 由约 3.2s 优化⾄ 2.5s，⾸屏资源压缩后体积下降约 26%

一、为什么要优化

- 你为什么关注首屏加载？
> 首屏是用户进入后的第一体验，直接影响跳出率和转化；而我的项目是内容展示型页面，LCP 越快越有利于留住用户，所以把首屏链路作为优化重点。

- 优化前首屏慢在哪里？
- 首屏需要加载的资源过多，主包偏大；一些非首屏路由的组件也跟着打进首屏包里；图片和静态资源没有做到按需加载

二、具体做了哪些优化
- 问法：你说的“路由级懒加载”怎么做？
>答法：把路由对应的页面组件改为异步 import()，让非首屏页面不进入首包，首屏只加载必须的页面与依赖。

- 问法：Vite 自动分包体现在哪？
> 答法：Vite 在构建时会把第三方依赖拆成 vendor 包，我这次还基于分析报告做了手动分包（element-plus、editorjs、three/vanta、lodash-es 拆成独立 chunk），让首屏只加载必要依赖，其它重包按路由或功能延后加载。这个拆分结果可以在 vite build 的产物列表和分析报告里直观看到。

- 问法：静态资源按需加载具体做了什么？
> 答法：图片资源优先加载首屏必要内容，非首屏或非关键图片延后加载（虚拟列表或图片懒加载），减少首屏负担。

三、你是怎么测数据的？
- LCP 3.2s → 2.5s 怎么测的？
> 用 Chrome DevTools 的 Performance/Lighthouse 测量同一页面、同一网络条件（例如 Fast 3G 或固定的网络限速），取多次平均值对比。


- 首屏资源体积下降 26% 怎么算？
> 用 vite build 后查看产物大小（dist 里首屏相关 chunk 的体积），对比优化前后的 gzip 或 brotli 体积，按比例计算。

> 我做首屏优化的原因是首屏是用户第一感知，对留存和体验影响最大。优化前首包偏大，非首屏路由和静态资源被提前加载，导致首屏 LCP 偏慢。
我主要做了三件事：
>1. 路由级懒加载，把非首屏页面改成异步 import()，缩小首包；
>2. 基于 Vite 分包和分析报告做手动拆分，把 element-plus、editorjs、three/vanta、lodash-es 这些重依赖单独分包，避免跟首屏业务混在一起；
>3. 静态资源按需加载，非关键图片延后加载，减少首屏带宽占用。
>
>效果上，LCP 从约 3.2s 降到 2.5s，首屏资源体积减少约 26%。数据来自 DevTools/Lighthouse 固定网络条件下多次测量取平均值，首屏资源体积是结合 vite build 产物体积对比。
>(代码分割、图片懒加载、虚拟列表)

##### •	基于 Node.js + Express 设计 RESTful API，结合 JWT 实现鉴权与接⼝权限控制，⽀持博客⼴场、多⽤户内容发布、评论点赞等核⼼业务闭环

- 你主要是前端，为什么要写 Node/Express？

>项目希望形成完整业务闭环（登录、发帖、评论、点赞），我在ai的帮助下用 Node/Express 做最小可用后端，保证接口规范统一、权限可控，前端联调效率高


- RESTful API是什么？ 怎么设计的？

> 一种 API 设计规范，用 HTTP 方法 + URL 来表达对资源的操作。
> 按资源划分，比如 articles、comments、likes、users，用 GET/POST/PUT/DELETE 表达读写删除；状态码用 200/201/401/403/404；响应结构统一（例如 status/message/data），便于前端统一拦截处理。

- JWT 具体怎么做的？
> 登录成功签发 token，前端请求拦截器统一加 Authorization；后端用中间件验证 token 合法性，解析用户信息，再做权限判断（如只能删除自己的帖子/评论，管理员可删除所有）。

- token 为什么放在localstorage 里，一般不是放在 cookie 里的吗？
> 项目背景
这是我独立完成的前后端项目，整体目标是先跑通完整业务闭环。
>
>自我评估
后端安全能力和投入时间有限，我优先保证功能完整、联调顺畅。
>
>技术选择
因此用 JWT 搭配 Pinia 存储，并持久化到 localStorage，保证刷新后仍保持登录态，前端拦截器统一加 Authorization。
>
>最终结果
方案简单可控、实现成本低，适合个人项目阶段；如果后续有更高安全要求，我会升级为 HttpOnly Cookie + CSRF 防护

- 接口权限控制到什么粒度？
 > 至少做到“未登录不可操作、登录后可操作自己的资源、管理员有管理权限”；比如删除帖子/评论会校验 user_id 或 role。


- 博客广场、多用户发布、评论点赞怎么串起来？
> 核心是用户 → 帖子/文章 → 评论/点赞 的数据关系。发布内容后在广场展示，评论/点赞只影响相关资源；后端做统计字段（点赞数、评论数）方便前端列表展示

##### 1. 在“AI Resume Analyzer”项目中，你具体用Zustand管理了哪些全局状态？并解释为什么选择Zustand而不是React自带的Context API或useReducer。

>在 AI Resume Analyzer 里，我用 Zustand 管的全局状态主要是三类：
>
>系统状态
>
>isLoading、error：全局加载与错误态
认证状态
auth.user、auth.isAuthenticated
signIn / signOut / refreshUser / checkAuthStatus 这些认证动作
>更少样板代码：不需要写很多 Provider、action type、switch，开发效率更高。
订阅粒度更细：组件可以只订阅自己关心的 slice，减少不必要重渲染。
业务更贴合：我们这个项目有大量“异步副作用 + 跨页面共享能力”（认证、文件、AI、KV），Zustand 把“状态 + 动作”放在一个轻量 store 里更清晰。
可维护性更好：相比 Context 在状态变大时容易臃肿，Zustand 更容易按模块拆分和扩展。

面试回答模板
》状态管理工具的选型我一般从几个维度考虑：状态复杂度、更新频率、性能要求以及团队规模。
如果是简单的跨组件状态，比如主题、用户信息，我会优先用 Context，因为它足够轻量
如果是中等复杂度业务，我更倾向用 Zustand，它的 API 简洁、性能好，而且支持按需订阅，适合大多数前端业务场景
但如果是大型项目或者状态逻辑非常复杂，比如有严格的数据流和副作用管理需求，我会选择 Redux Toolkit，它在可维护性、调试能力和团队协作上更有优势。
实际项目中我一般不会只用一种方案，而是 Context + Zustand 或 Redux 组合使用，根据不同状态类型分层管理。


##### 2. scss
SCSS 本质上是 CSS 的预处理器，主要解决原生 CSS 在大型项目里可维护性差的问题。它支持嵌套、变量、复用和模块化，让样式从“堆规则”变成“可组织的代码”

>我把 SCSS 当成工程化 CSS 来用，核心价值是统一和可维护。
第一点是全局变量统一主题和组件库风格。我们项目会先在全局层定义主题色、文字色、状态色这些设计令牌，然后页面和组件都复用这一套，不会每个人自己写一套颜色。这样做的好处是，风格一致、改版成本低，比如后面要从一种主题换到另一种主题，改变量就能全局生效，不需要逐页找样式。对组件库也一样，可以基于统一变量做二次定制，保证业务页面和组件库视觉是同一套语言。
>
>第二点是用嵌套加 scoped 提升可读性并隔离样式。SCSS 的嵌套可以把父子结构、状态样式、伪类写在一起，读起来就是组件结构本身，维护时定位很快。再配合 scoped，样式作用域默认限制在当前组件，基本不会出现一个页面改样式把另一个页面误伤的问题。对于多人协作，这点很重要，能明显减少样式冲突和回归问题。



#### 3. ai 对话

##### 2. Markdown 增量渲染中如何处理粘包、半包？ 
> 半包/黏包是在流解析层解决的
> 半包:
> - SSE 读到的字节先拼进 SSE 缓冲区里
> - 按 \n 拆行时，最后可能是不完整的一行，就将它取出并保留在缓冲区里，等下一次数据再拼上继续解析。
> 这样就能解决不完整的数据不会被误解析
> 粘包：
> - 粘包的时候，一次读到的内容里会带好几行 data:。我先按行拆开，再逐行处理，每一行都单独解析成 JSON、提取 delta.content，这样多条数据就不会混在一起
> - 解析出的内容再进渲染缓冲区做节流输出，避免频繁渲染。

##### 3. 流式输出（SSE）下 Markdown 不完整怎么处理？
> “我项目里‘双缓冲 + 宽松解析’是这样落地的：
>
>第一层缓冲（网络层）：SSE 数据先进入 SSE 缓冲区（字符串缓冲），按 \n 拆行；拆不完整的一行就先留着，等下一次数据再拼，这样能解决半包/粘包。
第二层缓冲（渲染层）：解析出 content 后先放进 渲染缓冲区，再用定时器每 50ms 吐一点给 UI，控制刷新频率，避免卡顿。
宽松解析：上层只是把新片段拼到 message.content，不做 AST 增量解析；真正的 Markdown 解析交给 ReactMarkdown + remark-gfm（你这里用的是 remarkGfm），它对不完整的 Markdown 有一定容错，流式阶段也能显示。”

##### 4. SSE、Fetch ReadableStream、WebSocket 的对比？

SSE

- 方向：服务端单向推送（server → client）
- 优点：基于 HTTP，穿透代理/负载均衡好，浏览器原生支持
- 缺点：只能单向，二进制不友好
- 适合：LLM 流式输出、通知类推

Fetch + ReadableStream

- 方向：请求发起后，服务端持续返回流
- 优点：实现简单、兼容性好、可控（你项目就是这种）
- 缺点：语义上不是“持续连接”，需要服务端支持分块输出
- 适合：一次请求的流式响应（LLM 逐 token 输出）

WebSocket

- 方向：全双工（双向实时）
- 优点：真正的实时双向通信，延迟低
- 缺点：需要 WS 连接管理、心跳、断线重连，运维更复杂
- 适合：双向实时场景（协作编辑、IM、游戏）

一句话选型：

只要单向流式输出 → SSE / Fetch 流就够了
需要双向实时交互 → WebSocket

##### 5. 
- SSE：描述的是服务器往客户端推送数据的协议/格式（server → client）。
- Fetch + ReadableStream：描述的是客户端如何读取这条流式响应（client 侧读取方式）。

它们怎么配合：
1. 前端用 fetch 发起请求；
2. 后端用 SSE 格式不断写 data: ...\n\n；
3. 前端用 ReadableStream 逐块读取，再按 SSE 规则拆包解析。

所以：SSE 是服务端输出格式，ReadableStream 是前端读取手段，两者正好配套。
