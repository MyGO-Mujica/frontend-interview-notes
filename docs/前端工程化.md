# 前段工程化

### 常见面试题

#### 1. Webpack
核心思想：
“**一切皆模块**” + “**提前打包**”

它的思路是：
- 开发前
- 把整个项目
- 一次性打包成浏览器可直接运行的文件

为什么慢？
不是“它写得差”，而是设计取舍：
- 启动时要分析整个项目
- 文件一多，依赖一复杂，成本立刻上来
- 即使你只改了一个文件，也可能触发较大的重建

Webpack 是“**稳定、通用、可控**”的代表，没有什么最优解，在工程上你需要做出取舍。

#### 2. Vite

Vite 的一句话定位：
**利用浏览器原生 `ES Module`，把“打包”这件事拆开**

1. **Vite 的 dev 模式做了什么**
你启动 Vite 时，它并没有做完整打包。
```js
浏览器请求 index.html
↓
index.html 里有 <script type="module">
↓
浏览器自己发起模块请求（import 链路）
↓
Vite 只在“被请求到的模块”上做处理
```
**重点来了**：
- 没有一次性扫描整个项目
- 没有完整打包
- 用多少模块，处理多少模块
这就是 Vite 启动快的根本原因。

故： 为什么 Vite 启动快？
> Vite 在开发模式下利用浏览器原生 ES Module 能力，不做全量打包，只在模块被实际请求时按需处理，因此避免了传统工具在启动时构建完整依赖图的开销，从而显著提升冷启动速度。


##### Vite 与 Webpack 的不同对比，原理理解
最核心的区别：开发环境的处理方式不同
Webpack：先打包再启动
```
所有模块 → 分析依赖 → 打包成 bundle → 启动开发服务器
项目越大，启动越慢
```

Vite：先启动再按需加载
```
直接启动服务器 → 浏览器请求哪个文件才处理哪个
利用浏览器原生 ES Module，不需要打包
```

**热更新**

Webpack：
-   → 从入口重新分析依赖
    → 重新打包所有相关模块
    → 项目越大，相关模块越多，越慢

Vite：
- → 直接找到这个文件对应的模块
  → 只更新这一个模块
  → 和项目大小无关，始终很快

> Vite 热更新通过 WebSocket 实现。启动时 Vite 和浏览器建立 WebSocket 连接，文件改动时 Vite 检测到变化，通过 WebSocket 通知浏览器哪个模块更新了，浏览器重新请求这个模块的最新内容，只替换这个模块，不刷新整个页面，所以页面状态、表单输入都会保留。如果改动影响范围太大无法局部替换，才会整页刷新兜底。


#### 对比表

|            | Webpack          | Vite                     |
| ---------- | ---------------- | ------------------------ |
| 开发启动   | 慢，需要打包     | 快，按需处理             |
| 热更新     | 项目越大越慢     | 始终很快                 |
| 生产打包   | 自己的打包器     | Rollup                   |
| 配置复杂度 | 复杂             | 简单，开箱即用           |
| 兼容性     | 好，支持旧浏览器 | 依赖现代浏览器 ES Module |
| 生态       | 成熟，插件多     | 较新，插件在增长         |
