# HTTP

### 常见面试问题

#### 1. URL

##### 1. URL输入浏览器之后会发生什么？

整体流程：

> **解析 URL → DNS 解析 → 建立连接 → 发送 HTTP 请求 → 服务器处理 → 返回响应 → 浏览器解析渲染 → 页面显示**

详细拆解

1. URL 解析
   浏览器会先解析你输入的 URL，比如：

```
https://www.example.com:443/path?a=1#hash
```

解析出：

- 协议（HTTPS）
- 域名（www.example.com）
- 端口（443）
- 路径（/path）
- 查询参数（a=1）
- hash（`#hash`，不参与请求）

同时浏览器还会做：

- 检查是否是合法 URL
- 补全协议（你输入 google.com 时自动加 https）

2.  DNS 解析（域名 → IP）
    **查询顺序：**

          1. 浏览器缓存
          2. 操作系统缓存
          3. hosts 文件
          4. 本地 DNS 服务器
          5. 根 DNS → 顶级域 → 权威 DNS

    最终得到 IP，比如：www.example.com → 93.184.216.34
    
    >DNS 解析就是将域名转为 IP 的过程。首先浏览器先查自身缓存，没有就查看本地系统缓存和 hosts 文件；如果还没有，就向本地 DNS 服务器发起递归查询；本地 DNS 依次访问根域名服务器、顶级域名服务器、权威域名服务器，最终拿到目标 IP 并逐级返回，同时做缓存，浏览器拿到 IP 后，就可以和服务器建立通信

3.  建立连接（TCP / TLS）
    (1) TCP 三次握手
    客户端和服务器建立连接：
    - SYN（客户端）
    - SYN + ACK（服务器）
    - ACK（客户端）

目的是确保双方通信正常

（2）如果是 HTTPS，还要 TLS 握手
做的事情：验证服务器证书、协商加密算法、生成会话密钥
握手之后通信都是加密的

4. 发送 HTTP 请求
   浏览器构造请求报文

```
GET /path HTTP/1.1
Host: www.example.com
User-Agent: Chrome
Cookie: xxx
```

可能涉及：

- Cookie（登录态）
- 缓存（If-None-Match / If-Modified-Since

5. 服务器处理请求

6. 返回 HTTP 响应

7. 浏览器解析与渲染（重点）
   1. 解析 HTML → DOM 树
   2. 解析 CSS → CSSOM 树
   3. 合并 → DOM + CSSOM 合并 生成渲染树 Render Tree
   4. 布局（Layout / Reflow）
   5. 分层（Layer）
   6. 绘制（Paint）
   7. 合成（Composite）

8. JS 执行（阻塞点）
   注意：

- JS 会阻塞 HTML 解析
- CSS 会阻塞渲染（但不阻塞 DOM 解析）

所以才有：

- `defer`
- `async`

9.  页面加载完成

**总结：**

> 用户输入 URL 后，浏览器会先解析 URL，然后通过 DNS 解析获取服务器 IP，接着通过 TCP（三次握手）建立连接，如果是 HTTPS 还会进行 TLS 握手。连接建立后浏览器发送 HTTP 请求，服务器处理后返回响应。浏览器拿到 HTML 后会解析生成 DOM 树，同时解析 CSS 生成 CSSOM，再合成渲染树，经过布局和绘制最终展示页面。在这个过程中，JS 可能会阻塞解析，同时还涉及缓存、重定向和网络优化等机制。


##### 2. tcp为什么需要三次握手 与四次挥手
![三次握手示例](https://images.openai.com/static-rsc-4/K6wAmvfPmJk4t6_AXbka8sRHSwZ-1QKE2guONxPXxAdkfs03SU6tCIjtKk1aiVwE0xuFxHlJgtrU9As9qtYiGnvXgdZsC9gQTZjY0uCa-vmehEtnDrRTHlxsskd_pzIRkyRNOpJYUCKA9gCFkCOFDcR7j9Ri3_LLs14rpUoY8Lg?purpose=inline)

四次挥手实例
```js
客户端                    服务端
  |                         |
  |──── FIN ──────────────→ |  第一次：我要断了
  |                         |
  |←─── ACK ─────────────── |  第二次：好的我知道了
  |                         |  （服务端可能还有数据没发完）
  |←─── FIN ─────────────── |  第三次：我也发完了，我也要断了
  |                         |
  |──── ACK ──────────────→ |  第四次：好的，再见
  |                         |
```


**TCP 需要三次握手，是为了：**

1. 确认双方的收发能力正常
2. 同步双方的初始序列号（ISN）
3. 避免历史连接（旧报文）干扰新连接


**为什么挥手多了一次**
> 挥手时客户端说要断，服务端可能还有数据没发完，必须先单独回 ACK，等数据发完再单独发 FIN，多了一次。


#### 2. 浏览器解析与渲染
   1. 解析 HTML → DOM 树
   2. 解析 CSS → CSSOM 树
   3. 合并 → DOM + CSSOM 合并 生成渲染树 Render Tree
   4. 布局（Layout / Reflow）
   5. 分层（Layer）
   6. 绘制（Paint）
   7. 合成（Composite）


- 渲染阻塞问题
  - CSS 阻塞渲染：CSSOM 未构建完成前，页面不会渲染，会出现白屏；
  - JS 阻塞 DOM 解析 & 渲染：JS 执行会暂停 HTML 解析，因为 JS 可修改 DOM/CSSOM；
✅ 优化：给 script 加async/defer、将 script 放 body 底部。
- 重排（Reflow） vs 重绘（Repaint）
  - 重排：元素几何属性改变（宽高、位置、增删节点、窗口 resize）→ 触发完整渲染流程，性能消耗大；
  - 重绘：仅样式改变（颜色、背景）→ 跳过布局，只重新绘制，消耗较小；
  - 合成：transform/opacity → 跳过重排 + 重绘，仅合成图层，性能最优。
- 减少重排重绘的优化手段
  - 批量修改 DOM（离线操作 DOM，如先隐藏再修改）；
  - 使用 transform/opacity 实现动画

##### 1. 重排 vs 重绘
**重绘性能远比重排好**，重排开销极大，是前端性能优化里最需要避免的操作。
```
HTML → 解析 DOM
      ↓
CSS  → 解析 CSSOM
      ↓
   合并 → Render Tree（渲染树）
      ↓
   Layout（布局/排版）← 这里产生"重排"
      ↓
   Paint（绘制）← 这里产生"重绘"
      ↓
   Composite（合成）
      ↓
   显示到屏幕
```
1. 重排（Reflow / Layout）
浏览器重新计算元素的几何信息（位置、宽高、大小、布局结构），重新排布页面元素。
2. 重绘（Repaint / Paint）
元素几何信息不变，只重新绘制元素的视觉样式（颜色、背景、阴影等）。

>重排（Reflow） 是指浏览器重新计算元素的几何属性，比如宽高、位置、margin 这些，然后重新生成渲染树和布局。它的代价最高，可能触发多个元素的重新计算,损耗大量性能。
>
>重绘（Repaint） 是指只改变元素的视觉样式，比如颜色、背景色、阴影这些，但不涉及布局变化。它的代价相对较低。

优化重排：
- CSS class 来批量改样式
- 是优先用 transform 和 opacity 来做动画。


##### 2. Script放在html里面会阻塞html渲染吗,放哪里才不会阻塞?
**会阻塞**
浏览器解析 HTML 是从上到下的，遇到 `<script>` 标签时：
```
解析HTML → 遇到script → 停下来 → 下载JS → 执行JS → 继续解析HTML
```
因为 JS 可能操作 DOM（比如 document.write），所以浏览器不敢继续往下解析，必须等 JS 执行完。

现代解决方案：defer 和 async
```js
<head>
  <script defer src="./index.js"></script>
  <script async src="./index.js"></script>
</head>
```

> script 放在 head 里会阻塞 HTML 渲染，因为浏览器从上到下解析 HTML，遇到 script 就得停下来等 JS 下载和执行完，用户会感觉明显白屏。
传统做法是把 script 放在 body 最底部，这样 DOM 已经解析完了再加载 JS，不影响页面显示。
现代方案是用 defer 或 async。defer 是并行下载 JS，但等 HTML 完全解析完再执行，而且多个 defer 脚本会按顺序执行，适合依赖 DOM 的脚本。async 也是并行下载，但下载完就立即执行，不保证顺序，适合独立的脚本比如埋点统计。
一般项目里推荐用 defer，既不阻塞解析，又能保证执行顺序。

##### 3.浏览器缓存
每次请求
```
用户请求 → DNS 解析 → TCP 连接 → 发送请求 → 服务器处理 → 返回数据 → 渲染
```
缓存的目标：让浏览器直接用本地存的资源，**跳过网络请求**。

有两大缓存
- 强缓存  （不发请求，直接用本地缓存）

   HTTP/1.0 强缓存响应头头部字段用 `Expires`，HTTP/1.1响应头头部字段用 `Cache-Control`

- 协商缓存
 协商缓存用 `Last-Modified/If-Modified-Since` 和 `ETag/If-None-Match`
 
 1. 第一组 基于修改时间
 
    | 类型       | 头部字段            | 说明                                              |
    | ---------- | ------------------- | ------------------------------------------------- |
    | **响应头** | `Last-Modified`     | 服务器返回：资源的最后修改时间                    |
    | **请求头** | `If-Modified-Since` | 浏览器下次请求带上：上次缓存的 `Last-Modified` 值 |
 
 2. 基于哈希（更精确，优先级更高）
 
    | 类型       | 头部字段        | 说明                                           |
    | ---------- | --------------- | ---------------------------------------------- |
    | **响应头** | `ETag`          | 服务器返回：资源的唯一标识（通常是文件哈希值） |
    | **请求头** | `If-None-Match` | 浏览器下次请求带上：上次缓存的 `ETag` 值       |

```
浏览器发请求
      │
      ▼
① 强缓存：本地有没有？过期了吗？
      │
  没过期 ──────────────────→ 直接用本地资源（200 from cache）
      │
  过期了
      │
      ▼
② 协商缓存：问服务器，资源有没有变？
      │
  没变 ────────────────────→ 服务器返回 304，浏览器用本地资源
      │
  变了
      │
      ▼
  服务器返回新资源（200）
```

#### 4. 前端有哪些本地存储方式？
📚 核心知识点
1. Cookie

- 大小限制：4KB
- 每次请求会自动携带到服务端
- 可设置 `expires / max-age` 控制过期时间
- 属性：`HttpOnly`、`Secure`、`SameSite`
- 用途：身份认证、会话管理

2.  localStorage

- 大小限制：5MB 左右
- 永久存储，不主动清除则一直存在
- 同源共享，适合存用户偏好、token 等持久化数据
- API：`setItem / getItem / removeItem / clear`

3. sessionStorage

- 大小限制：5MB 左右
- 会话级，关闭标签页即清除,刷新不会消失
- 不同标签页之间不共享
- 用途：表单草稿、一次性数据

##### 1. 场景题： 我有 A、B 两个页面，同一域名，我在 A 页面设置了一个 local storage 并修改了，B 页面能感知到吗？
能感知到，通过 `storage` 事件
- 同源页面之间，`localStorage` 数据是共享的，A 页面修改了，B 页面可以通过监听 `storage` 事件感知到变化
- `storage` 事件不会在当前页面触发，只会在其他同源页面触发

#### 3. http状态码
五大类

| 范围 | 类型       | 记忆方式     |
| ---- | ---------- | ------------ |
| 1xx  | 信息       | 继续处理中   |
| 2xx  | 成功       | 没问题 ✅     |
| 3xx  | 重定向     | 去别处找 🔀   |
| 4xx  | 客户端错误 | 你的锅 ❌     |
| 5xx  | 服务端错误 | 服务器的锅 💥 |

2xx 成功
- `200 OK`：请求成功
- `201 Created`：资源创建成功（POST 新增数据）
- `204 No Content`：成功但没有返回内容（DELETE 常用）

3xx 重定向
- `301 Moved Permanently`：永久重定向，浏览器会缓存新地址
- `302 Found`：临时重定向
- `304 Not Modified`：协商缓存命中，资源未变，用本地缓存

4xx 客户端错误
- `400 Bad Request`：请求参数有问题
- `401 Unauthorized`：未登录 / token 失效
- `403 Forbidden`：已登录但没权限
- `404 Not Found`：资源不存在

5xx 服务端错误
- `500 Internal Server Error`：服务器内部报错
- `502 Bad Gateway`：网关错误，上游服务挂了
- `503 Service Unavailable`：服务不可用，通常是在维护或过载

#### 4. HTTP 1.0/1.1/2/3的区别是什么？

📚 核心知识点
HTTP/1.0

- 每次请求都要新建 TCP 连接，用完就断
- 没有 keep-alive，并发性能很差

----

HTTP/1.1

- 默认开启 keep-alive，连接可复用，不用每次重新建连接
- 但同一连接请求还是串行，有队头阻塞, 
- 支持分块传输、缓存控制

-----

HTTP/2

- 多路复用：一个连接并行发多个请求，解决队头阻塞
- header 压缩：HPACK 算法，不重复传相同 header
- 二进制分帧：比文本传输更高效
- 服务器推送：主动推资源给客户端
- 缺点：底层 TCP 层仍有队头阻塞（丢包时整个连接等待）

-----

HTTP/3

- 抛弃 TCP，改用 QUIC 协议（基于 UDP）
- 彻底解决 TCP 层队头阻塞
- 连接建立更快：QUIC 把 TCP 握手和 TLS 握手合并，从 3次 - RTT 降到 1次
- 支持连接迁移：切换网络（WiFi → 4G）不断连

#### 对比表

|             | 1.0  | 1.1  | 2          | 3        |
| ----------- | ---- | ---- | ---------- | -------- |
| 连接复用    | ❌    | ✅    | ✅          | ✅        |
| 队头阻塞    | 有   | 有   | 应用层解决 | 彻底解决 |
| 传输格式    | 文本 | 文本 | 二进制     | 二进制   |
| 底层协议    | TCP  | TCP  | TCP        | QUIC/UDP |
| header 压缩 | ❌    | ❌    | ✅          | ✅        |
| 服务器推送  | ❌    | ❌    | ✅          | ✅        |

>HTTP 四个版本核心区别是：
1.0 每次请求都要新建 TCP 连接，用完就断，性能很差。
1.1 加入了长连接 keep-alive，连接可以复用，但同一连接的请求还是串行的，有队头阻塞问题。
2 引入多路复用，一个连接可以并行发多个请求，解决了应用层的队头阻塞；还有 header 压缩和二进制传输。但底层还是 TCP，丢包时 TCP 层的队头阻塞依然存在。
3 直接换掉了 TCP，用基于 UDP 的 QUIC 协议，彻底解决队头阻塞；连接建立也更快，把 TCP 握手和 TLS 握手合并，大幅减少握手时间；还支持连接迁移，切换网络不会断连。


##### 1. HTTP 1.1 的长连接，它是指那个队头阻塞，它是指请求发不出去，还是响应的时候有堵塞？
> HTTP/1.1 的队头阻塞是发生在请求发送阶段。虽然长连接可以复用 TCP，但同一个连接上请求必须串行，也就是上一个请求的响应没回来，下一个请求就不能发出去，被堵在队列里了。
浏览器的解决方式是对同一域名开多个 TCP 连接来变相并发，但 HTTP/2 才真正解决了这个问题，通过多路复用让多个请求可以在一个连接上并行，互不干扰。

##### 2. HTTP 2.0 的时候，它的多路复用，什么情况下才能复用一个 TCP 连接呢？

> HTTP/2 的多路复用，能复用同一个 TCP 连接最基本的条件就是同一个域名。如果是不同域名，但解析到同一个 IP，并且服务器的 TLS 证书同时覆盖了这两个域名，浏览器也可以把它们合并到一个连接上，这叫连接合并。
另外实际使用中，HTTP/2 基本只在 HTTPS 下才生效，不同端口或者不同 IP 的域名就没办法复用，得各自新建连接。
复用之后的工作方式是，多个请求在同一个连接上以不同的 Stream 并行传输，互不阻塞，解决了 HTTP/1.1 的队头阻塞问题。

#### 5. https和http的区别，它用了什么机制去保证安全 

📚 核心知识点

**核心区别**

|      | HTTP   | HTTPS             |
| ---- | ------ | ----------------- |
| 传输 | 明文   | 加密              |
| 端口 | 80     | 443               |
| 安全 | ❌      | ✅                 |
| 证书 | 不需要 | 需要 SSL/TLS 证书 |

HTTPS 用了哪些机制保证安全?


1. 第一是**加密**，分两个阶段：先用非对称加密协商对称密钥，协商过程类似颜色混合——双方各自保留私有颜色，只交换与公共颜色混合后的颜色，中间人拦截到混合色也倒推不出原始私有颜色，所以密钥传输是安全的。密钥协商好之后，后续通信改用对称加密，因为对称加密速度快，适合加密大量数据。

2. 第二是**身份验证**，光加密还不够，还要确认对面是不是真正的服务器。服务器提前向 CA 申请证书，CA 用自己的私钥对证书签名。浏览器收到证书后，用内置的 CA 公钥验证签名，签名对了说明证书可信，对面就是真实服务器，防止中间人冒充。

3. 第三是**完整性校验**，发送方对数据做哈希得到摘要一起发出去，接收方收到后重新算哈希和摘要对比，一致说明数据没被篡改，不一致立刻丢弃。哈希是单向的，改了数据哈希就变了，但无法从哈希反推原始数据。

追问：

https里对称加密的密钥会在客户端和服务端之间传输吗？
- 不会直接传输！ 对称密钥是双方各自计算出来的，不是一方生成后发给另一方。

客户端和服务端从未传输过最终密钥，为什么双方能独立计算出相同的对称密钥？背后的数学原理是什么？


#### 6. GET 与 POST 的区别
📚 核心知识点
- 语义设计
  - GET：语义为查询读取，纯只读操作，不修改服务器资源，用于获取数据。
  - POST：语义为提交写入，用于新增 / 修改 / 资源，会产生数据变更。

- 幂等性
 幂等：多次执行请求，最终效果和只执行一次完全一致
  - GET 是幂等的：重复刷新、多次请求无副作用，数据不会重复变更。
  - POST 不是幂等的：每次请求都可能产生副作用，比如重复提交订单、重复插入数据

- 缓存机制
  - GET：浏览器默认开启缓存（强缓存 + 协商缓存），会缓存响应结果，减少重复请求。
  - POST：浏览器默认不缓存，每次都会完整发起网络请求，因为请求可能改变数据
  
- 浏览器行为
  - GET：参数拼接在 URL 上，长度受浏览器限制；刷新页面会直接重发
  - POST：参数放在请求体里，理论上无大小限制；刷新页面浏览器会弹窗提示"是否重新提交表单"


#### 7.常见的http请求方法有哪些
1. GET：查询资源
- 作用：从服务器获取指定资源（查），是最常用的请求方法。

2. POST：新增资源
- 作用：向服务器提交数据，创建新资源（增），比如提交表单、上传文件。

3. PUT：全量更新资源
- 作用：用请求体的内容，完整替换服务器上的指定资源（全量改）。
- 核心特点：
  - ❌ 不安全（会修改数据）；
  - ✅ 幂等：多次执行结果相同（比如把 id=1 的用户名改成 "张三"，执行 1 次和 10 次结果一样）；
  要求请求体包含资源的完整信息，缺省字段会被置空 / 覆盖。

4. DELETE：删除资源
- 作用：删除服务器上的指定资源（删）。
- 核心特点：
  - ❌ 不安全；
  - ✅ 幂等：删除同一个资源 1 次和 10 次结果相同（资源已删除后再次删除，结果还是不存在）

#### 8. 如何保证 JWT 是安全的？JWT 应该放在哪里？
>JWT 安全性靠几点保证：签名机制防止 内容 被篡改，服务端用密钥验签，改了内容签名对不上直接拒绝；使用 HTTPS 防止传输过程被截获；设置过期时间，泄露了也会自动失效
存放位置推荐 HttpOnly Cookie，JS 无法读取，能防 XSS 攻击，配合 SameSite 属性也能防 CSRF。放 localStorage 的话容易被 XSS 脚本读走，安全性低一些。

#### 9. Vue Router / React Router 原理及调用的浏览器 API
Vue Router 和 React Router 的路由原理是一样的，都是两种模式，调用的浏览器 API 也相同

Hash 模式：

- URL 带 #，如 http://xxx.com/#/home
- 调用 API：hashchange 事件，监听 window.addEventListene('hashchange',  callback)
- 原理：# 后面内容变化不会触发页面刷新，浏览器通过 hashchange 事件通知 JS，路由库在回调里匹配路由渲染组件

History 模式：
- URL 正常，如 http://xxx.com/home
- 调用 API：
  - pushState(state, title, url)：添加一条历史记录，修改 URL 但不刷新页面
  - replaceState(state, title, url)：替换当前历史记录，修改 URL 但不刷新页面
  - popstate 事件：监听浏览器前进/后退，window.addEventListener('popstate', callback)

原理：pushState/replaceState 修改 URL 但不触发任何事件，所以路由库需要自己封装跳转方法（Vue Router 的 router.push、React Router 的 <Link>），内部调用 pushState 并手动触发路由匹配。浏览器的前进后退会触发 popstate，路由库在回调里匹配路由

关键区别：
| 维度| Hash 模式|History 模式|
| ---| ---| ---|		
|修改 URL|	location.hash = '/home'|	history.pushState(null, '', '/home')
监听变化	|hashchange 事件	|popstate 事件（前进后退）+ 手动触发（pushState 不触发事件）
服务端配置	|不需要	|需要配置所有路由回退 index.html
SEO	|不友好（#后内容不发给服务器）|	友好

> 两个路由库原理一样，都是两种模式。Hash 模式监听浏览器的 hashchange 事件，修改 location.hash 就能切换路由，不需要服务端配置。History 模式用 pushState 和 replaceState 修改 URL 但不刷新页面，浏览器的前进后退通过 popstate 事件监听。但 pushState 本身不触发任何事件，所以路由库要自己封装跳转方法，内部调用 pushState 后手动触发路由匹配。两个库的差异主要是 Vue Router 内置了导航守卫，React Router 没有；React Router 基于 Context 驱动路由状态，Vue Router 自己管理。

#### 10. 前端性能优化方案

1. 加载优化
做路由懒加载、图片懒加载，通过动态 import 按需引入代码，减小首屏打包体积，加快首屏渲染。

2. 网络与缓存优化
合理利用浏览器强缓存、协商缓存，减少重复请求；接口使用分页、节流防抖，避免多余请求。

3. 页面渲染优化
尽量减少重排和重绘，动画优先用 `transform`、`opacity`；长列表使用虚拟列表，只渲染可视区域 DOM，避免页面卡顿；批量操作 DOM，减少频繁样式修改。

4. React 框架优化
使用 `React.memo`、`useMemo`、`useCallback` 缓存数据和函数，避免组件不必要的重复渲染；组件合理拆分，同时清空定时器、事件监听，防止内存泄漏。
